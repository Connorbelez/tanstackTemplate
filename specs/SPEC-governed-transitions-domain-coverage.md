## PRD
### Purpose
FairLend needs a governed-transition layer that can survive two separate questions from an auditor:
1. What state did each mortgage, obligation, and renewal-intent entity occupy on any day?
2. Who caused each change, through what channel, with what supporting evidence, and how does that tie back to the money ledger?

The current stack is close, but not complete. `convex/engine/transition.ts` already persists transition results into `auditJournal`, and `convex/engine/hashChain.ts` forwards those rows into the auditTrail component. The mortgage and obligation machines also exist in `convex/engine/machines/mortgage.machine.ts` and `convex/engine/machines/obligation.machine.ts`. The gap is breadth and reconstruction quality: the domain events are not yet serialized as a canonical replay envelope, renewal intent is not governed, and the audit export path does not preserve enough provenance for legal-grade reconstruction.

This spec defines the missing behavior.

### Current State
| Area | Current behavior | Gap |
| --- | --- | --- |
| Mortgage lifecycle | `active -> delinquent -> defaulted -> collections -> written_off`, plus `active -> matured` in `convex/engine/machines/mortgage.machine.ts` | No canonical export contract for each transition, and no explicit renewal-intent lifecycle |
| Obligation lifecycle | `upcoming -> due -> overdue -> partially_settled -> settled/waived` in `convex/engine/machines/obligation.machine.ts` | No export-grade linkage to the parent mortgage and no full replay package |
| Scheduler coverage | `convex/payments/obligations/crons.ts` drives `BECAME_DUE` and `GRACE_PERIOD_EXPIRED` only | No first-class cron coverage for default, collections, write-off, maturity, or renewal deadlines |
| Audit persistence | `auditJournal` stores actor, actorType, channel, ip, sessionId, machineVersion, payload, and state change | `convex/engine/hashChain.ts` only forwards a reduced metadata payload, so reconstruction loses provenance detail |
| Renewal intent | `lenderRenewalIntent` exists in schema and `EntityType`, but is not in `convex/engine/machines/registry.ts` | No governed renewal-intent machine or export contract |
| Reconstruction | `auditJournal` and `auditTrail` are tamper-evident | There is no canonical replay package that a human or program can ingest to rebuild the timeline and final state |

### Scope Boundary
This work must cover:
- Mortgage lifecycle states and events.
- Obligation lifecycle states and events.
- Default, collections, write-off, maturity, and renewal-intent evidence.
- Reconstruction-grade export of event history, provenance, and linked downstream effects.
- Composition with ledger and simulation so state replay and money replay can be correlated.

This work must not promise:
- New economics or servicing policy.
- A full renewal execution system if the corresponding machine is not yet implemented.
- UI work beyond what is needed to expose the export or replay surface.
- Replacement of the money ledger.

If renewal execution is not fully modeled by the time this spec is implemented, the supported boundary is:
- `lenderRenewalIntent` is governed and exported as intent evidence.
- Renewal execution, repricing, extension, or refi conversion remains out of scope.
- Simulation should cap at 11 months unless renewal and default handling are fully implemented end to end.

### User Stories
- As an auditor, I can export a mortgage timeline and rebuild the exact terminal state from the exported rows alone.
- As an operator, I can see which actor, session, channel, and request caused a default, collections step, or write-off.
- As a compliance reviewer, I can verify that the hash chain and the export package agree on ordering, payload, and provenance.
- As a simulator, I can replay a time-ordered command stream and prove that the final mortgage and obligation states match production.
- As a finance reviewer, I can correlate each governing transition with the ledger entries or payment effects it triggered.

### Functional Requirements
- Every governed transition must serialize to a canonical envelope that is replayable without reading the live tables first.
- Every envelope must include full provenance: actorId, actorType, channel, ip, sessionId, requestId, correlationId, originSystem, and legalEntityId where known.
- Every transition row must preserve the previous state, new state, outcome, machine version, effective timestamp, and idempotency context.
- Every transition row must link to any downstream ledger entry IDs, dispersal IDs, or obligation IDs generated by scheduled effects.
- Every default, collections, write-off, maturity, and renewal-intent event must be auditable even if it is a no-op state change in the machine.
- Export output must be deterministic for the same source rows, ordering, and as-of timestamp.
- Reconstruction must work from an export package, from live tables, and from the hash chain, and the three results must agree.

### Non-Functional Requirements
- Export and replay must be deterministic and idempotent.
- Audit records must remain append-only.
- Legacy rows must remain readable after schema extensions.
- Export packages must be checksum-verifiable and safe to archive.
- Reconstruction helpers must be explicit about unsupported cases rather than guessing.

### Acceptance Criteria
- A single mortgage can be exported, replayed, and reconstructed to the exact same final state from the generated package.
- A default-to-collections-to-write-off path is fully visible in the audit export, including actor and scheduler provenance.
- An obligation paid in partial steps can be replayed from the export and reaches the same settled or partially-settled state as production.
- A renewal-intent row can be exported and re-read even if renewal execution is not implemented.
- The export package includes enough metadata to prove which downstream ledger and dispersal records were created by each transition.
- The hash chain verifies the same ordered event stream that the export package contains.

## TDD
### Architecture Overview
This spec should evolve the existing governed-transition stack rather than replace it.

Primary touch points:
- `convex/engine/transition.ts` becomes the canonical transition writer and replay envelope builder.
- `convex/engine/machines/registry.ts` gains any new governed machines, including renewal-intent coverage if the product chooses to govern it.
- `convex/engine/machines/mortgage.machine.ts` grows the missing lifecycle breadth for defaults, collections, write-off, maturity, and renewal-related events.
- `convex/engine/machines/obligation.machine.ts` keeps the payment lifecycle but must emit replay-friendly payload metadata for downstream reconstruction.
- `convex/engine/effects/obligation.ts` remains the cross-entity bridge from obligation transitions to mortgage transitions and ledger/dispersal side effects.
- `convex/payments/obligations/crons.ts` becomes the scheduler entry point for due, overdue, default, collections, write-off, maturity, and renewal-deadline checks.
- `convex/engine/hashChain.ts` and `convex/components/auditTrail/lib.ts` must preserve the full canonical metadata, not a reduced subset.

### Canonical Event Envelope
Introduce a normalized transition envelope that is the single source for replay, export, and hash-chaining.

Recommended fields:
```ts
interface GovernedTransitionEnvelope {
  eventId: string;
  entityType: "mortgage" | "obligation" | "lenderRenewalIntent" | string;
  entityId: string;
  eventType: string;
  previousState: string;
  newState: string;
  outcome: "transitioned" | "rejected";
  timestamp: number;
  effectiveDate?: string;
  machineVersion: string;
  payload?: Record<string, unknown>;
  causedBy?: string;
  idempotencyKey?: string;
  provenance: {
    actorId: string;
    actorType?: string;
    channel: string;
    ip?: string;
    sessionId?: string;
    requestId?: string;
    correlationId?: string;
    originSystem?: string;
    legalEntityId?: string;
  };
  effectsScheduled: string[];
  linkedRecords?: {
    ledgerJournalEntryIds?: string[];
    dispersalEntryIds?: string[];
    obligationIds?: string[];
  };
}
```

The implementation can keep storing rows in `auditJournal`, but the row shape and the hash-chain payload must be derived from this envelope rather than an abbreviated subset.

### Data Model Changes
Add only additive schema changes first.

1. Extend `auditJournal` with optional provenance fields that are not yet guaranteed today: `requestId`, `correlationId`, `originSystem`, `legalEntityId`, and `eventId` or equivalent stable replay key.
2. Keep `actorId`, `actorType`, `channel`, `ip`, `sessionId`, and `machineVersion` indexed and queryable.
3. Add a replay/export tracking table such as `transitionExports` or `reconstructionRuns` with:
   - `scopeType`
   - `scopeId`
   - `asOf`
   - `format`
   - `status`
   - `inputHash`
   - `outputHash`
   - `createdAt`
   - `createdBy`
4. If renewal intent becomes governed, add its state machine data to the same reconstruction strategy instead of inventing a separate export format.

### Machine Coverage
`convex/engine/machines/mortgage.machine.ts`
- Keep the current states as the baseline.
- Add explicit event names for the missing lifecycle breadth so the export can distinguish them from generic scheduler activity.
- Treat the existing `DEFAULT_THRESHOLD_REACHED`, `COLLECTIONS_INITIATED`, and `WRITE_OFF_APPROVED` events as first-class exported events, not hidden implementation details.
- Add mortgage-level transitions or aliases for maturity and renewal-intent milestones if the product needs them to be visible in audit output.

`convex/engine/machines/obligation.machine.ts`
- Preserve the current payment lifecycle.
- Ensure `PAYMENT_APPLIED` exports the applied amount, settlement identifier, and enough reference data to correlate back to the money ledger or payment rail.
- Ensure `OBLIGATION_WAIVED` carries the approval reason and actor in the envelope, not only in the transition row.

`convex/engine/machines/registry.ts`
- Add a governed renewal-intent machine only if the team agrees that intent needs its own lifecycle.
- If that machine is added, the minimum lifecycle should support `draft -> signalled -> acknowledged -> expired/withdrawn`, with conversion/execution explicitly out of scope for this spec.

### Scheduler and Effect Contracts
`convex/payments/obligations/crons.ts`
- Split the cron responsibilities into explicit phases rather than only `BECAME_DUE` and `GRACE_PERIOD_EXPIRED`.
- Required scheduled checks:
  - due date arrival
  - grace-period expiration
  - default threshold evaluation
  - collections initiation eligibility
  - write-off eligibility
  - maturity date arrival
  - renewal-intent deadline or acknowledgment deadline
- Each cron phase must call the governed transition engine, not patch tables directly.

`convex/engine/effects/obligation.ts`
- Continue to forward obligation transitions to mortgage-level events.
- Add effect contracts for any new mortgage-level event names introduced above.
- Every effect must be idempotent and include the source provenance it received from the parent transition.
- If an effect creates ledger or dispersal entries, it must attach the created record IDs back onto the transition envelope or the export linkage table.

### Hash Chain and Export Pipeline
`convex/engine/hashChain.ts`
- Stop flattening the journal payload down to a reduced metadata blob.
- Insert the canonical envelope or a faithful JSON serialization of it into the auditTrail component.
- Preserve actorType, ip, sessionId, requestId, correlationId, originSystem, and legalEntityId where available.
- Hashing must cover the same canonical payload that the export package will emit, so hash verification and export verification are equivalent checks.

Recommended export package layout:
- `manifest.json` for scope, timestamp, checksum, and schema version.
- `events.csv` for ordered governed transitions.
- `states.csv` for optional day-level or as-of snapshots.
- `linkage.csv` for downstream ledger, dispersal, and obligation references.
- `hashes.csv` for chain verification and package integrity.

The export helper should support:
- `exportTransitionPackage(scopeType, scopeId, asOf?)`
- `reconstructFromExport(packagePath | parsedPackage)`
- `verifyExportPackage(packagePath | parsedPackage)`

### Replay and Reconstruction
Replay must be deterministic.

Algorithm requirements:
1. Load the canonical envelope rows in strict sequence order.
2. Reconstruct machine state from the serialized previous state and event payload.
3. Verify the live row matches the replayed state transition.
4. Apply linked side effects only as recorded references, never by re-running business logic during export replay.
5. Compare the reconstructed terminal state with the current live document state.
6. Compare the export checksum with the hash chain checksum.

The replay engine must fail closed if:
- a row is missing required provenance,
- a transition refers to an unknown machine version,
- a linked side-effect ID is missing when the row claims one exists,
- or an event cannot be deterministically rehydrated.

### Composition With Ledger and Simulation
This spec must compose with the money ledger and the simulation harness rather than duplicate them.

- Transition rows should reference ledger journal entries when a transition triggered money movement or payment allocation.
- The simulator should use the same governed transition surfaces as production so replay is comparable.
- If simulation drives lower-level ledger primitives for secondary trades, the exported event envelope must still point back to the originating simulation command and the resulting ledger entries.
- The reconstruction package should be able to explain both halves of the system: state transitions and money movements.

### Testing Strategy
Cover the implementation with layered tests:

1. Machine tests for every supported event/state path in mortgage and obligation machines.
2. Transition engine tests that validate canonical envelope creation, provenance propagation, and idempotency.
3. Hash-chain tests that prove the export payload and the auditTrail payload are byte-for-byte compatible at the serialized level.
4. Cron tests for due, overdue, default, collections, write-off, maturity, and renewal-deadline paths.
5. Effect tests that verify mortgage-level events are emitted from obligation-level changes and that linked record IDs are recorded.
6. Export/replay round-trip tests that rebuild a mortgage or obligation timeline from the export package and match live state.
7. Negative tests for missing provenance, unsupported machine versions, and partial renewal coverage.

### Rollout Gates
Use a staged rollout because this touches the canonical audit path.

1. Additive schema changes only, with legacy rows still readable.
2. Transition writer updated to populate the new envelope fields.
3. Hash-chain adapter updated to carry the canonical payload.
4. Export/replay tooling added and verified against a small fixture set.
5. Cron coverage expanded to the missing lifecycle phases.
6. Renewal intent either gets its own governed machine or is explicitly limited to export-only evidence.
7. Final gate: `bun check`, `bun typecheck`, `bunx convex codegen`, and the new round-trip tests all pass.

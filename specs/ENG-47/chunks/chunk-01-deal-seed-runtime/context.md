## Architecture Context
### SPEC 1.4 — Deal Closing
#### 3.1 Machine Shape
The deal machine uses XState v5's nested compound states. Each phase is a compound state with internal sub-states. Phase completion uses `type: "final"` on the last sub-state, triggering the parent's `onDone` transition to the next phase.

#### 3.2 machineContext Strategy
Minimal context: `{ dealId: string, reservationId?: string }`.
- `dealId` — the entity ID, standard GT pattern
- `reservationId` — set by the `reserveShares` effect on DEAL_LOCKED, used by `commitReservation` and `voidReservation` effects. Stored in context (not as a domain field) because it's machine-lifecycle state consumed by effects, not queryable data.

#### 4.2 Serialization Strategy
Store `status` as `v.string()` consistently across ALL governed entities. For compound states, use a dot-notation string:
```typescript
function serializeState(stateValue: StateValue): string {
  if (typeof stateValue === "string") return stateValue;
  const [region, subState] = Object.entries(stateValue)[0];
  if (typeof subState === "string") return `${region}.${subState}`;
  return `${region}.${serializeState(subState)}`;
}

function deserializeState(status: string): StateValue {
  if (!status.includes(".")) return status;
  const parts = status.split(".");
  let result: any = parts[parts.length - 1];
  for (let i = parts.length - 2; i >= 0; i--) {
    result = { [parts[i]]: result };
  }
  return result;
}
```

Implementation Plan · ENG-206

Dispersal → Disbursement Bridge

Convert eligible dispersal entries into outbound lender payout transfers. One transfer per entry, idempotent, pre-checked against the LENDER_PAYABLE balance gate, with effect-driven status lifecycle and daily admin alerts.

Estimate: 8 pts Priority: High Provider: mock_eft (Phase 1) 15 tasks · 3 chunks Files: 1 new · 2 modified
Section 01

The Problem

Before — No Bridge
1Borrower payment settles → obligation marked settled
↓
2Dispersal entries created per lender (pro-rata, fee deducted)
↓
3Entries sit in pending — forever
↓ dead end
4No mechanism to convert entries into outbound lender payments
5No cash ledger posting for lender payouts
6LENDER_PAYABLE balance accrues but never discharges
After — Bridge Active
1Entries accumulate in pending with hold period
↓
2Daily cron alerts admin: "N entries due across M lenders"
↓
3Admin triggers bridge → disbursement gate checks balance
↓
4Bridge creates one lender_dispersal_payout transfer per entry
↓
5Transfer confirms → LENDER_PAYOUT_SENT journal posted
↓
6Dispersal entry → disbursed with payoutDate

Section 08

Edge Cases

Scenario Expected Behavior
Duplicate bridge run Idempotency key prevents second transfer. Entry skipped.
Concurrent runs Second mutation hits idempotency check → returns existing. No double-spend.
Insufficient LENDER_PAYABLE Gate throws → entry stays pending → error in result. Others continue.
Entry status race Mutation re-reads entry. If no longer pending, skips.
Immediate provider confirm initiated → confirmed in one action. Entry becomes disbursed synchronously.
Async provider initiated → pending. Entry stays pending until webhook fires FUNDS_SETTLED.
Transfer failure Entry → failed. Admin can resetFailedEntry → re-run bridge.
Legacy entries (no hold period) Treated as immediately eligible.
Batch limit hit Remaining entries picked up on next trigger.
ENG-219: amount guard Bridge uses entry.amount as-is. Never recalculates.
Mock providers disabled areMockTransferProvidersEnabled() check prevents usage. Must configure real provider.

Section 11

Implementation Notes

All Blockers Resolved

ENG-199 (Cash Ledger Bridge), ENG-217 (Servicing Fee), ENG-216 (Granularity), ENG-220 (MockTransferProvider) — all Done.

Locked Decisions

Per-entry granularity. Pending → Disbursed directly. Admin-only Phase 1. Amount passthrough (ENG-219).

Critical — Do Not Recompute Amounts

Bridge MUST use entry.amount as-is. Recomputing at disbursement time violates ENG-219 if ownership changed since creation.

Mock Provider Gating

Phase 1 uses mock_eft (requires ENABLE_MOCK_PROVIDERS="true"). When VoPay sandbox arrives, eft_vopay drops in with zero bridge changes.

Downstream — ENG-219

ENG-219 is blocked by this issue. The no-recompute constraint is enforced by design here.

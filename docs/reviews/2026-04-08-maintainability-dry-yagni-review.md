## Executive Summary

The branch is moving in the right direction functionally, but maintainability debt is accumulating in three places: demo orchestration is writing directly into canonical payment state, collection-plan admin/workout code is growing as parallel monoliths, and test infrastructure is split across multiple overlapping bootstrap layers.

I did not find a single catastrophic maintainability blocker, but I did find several `High` issues that will make future payment/admin work more expensive than it needs to be. The strongest positive signal is that `convex/payments/transfers/mutations.ts` is already extracting some shared validation and provider-boundary logic; that pattern needs to spread to the rest of the branch.

## Severity-Ranked Findings

1. **High**: Demo-only orchestration is leaking directly into canonical payment state and will force product code to carry demo complexity.
   - Evidence:
     - `convex/demo/amps.ts:256-394` loads all mortgages, all plan entries, all attempts, and all workout plans, then filters in memory to derive demo scenario targets.
     - `convex/demo/amps.ts:469-607` writes demo-specific collection rules directly into `collectionRules`.
     - `convex/demo/amps.ts:611-740` seeds retry scenarios by inserting `collectionAttempts`, `transferRequests`, and patching `collectionPlanEntries` directly.
     - `convex/demo/amps.ts:782-934` orchestrates canonical admin and execution actions to shape demo state.
     - `convex/schema.ts:1848-2015` shows the canonical schema already carrying a large demo-only footprint.
   - Why this is a maintainability problem:
     - The demo module is no longer just reading canonical state; it is becoming a second orchestration layer for payments/admin behavior.
     - Every future change to collection rules, attempts, workout flows, or transfer state now has a second “demo compatibility” tax.
     - On a greenfield project with no prod data, this is exactly the sort of compatibility surface YAGNI says to avoid.

2. **High**: `collectionPlan/admin.ts` and `collectionPlan/workout.ts` are parallel monoliths with missing shared abstractions for read models, lifecycle guards, and action wrapping.
   - Evidence:
     - `convex/payments/collectionPlan/admin.ts:269-458` mixes row shaping, transfer summaries, reconciliation summaries, and nested row composition in one file.
     - `convex/payments/collectionPlan/admin.ts:530-912` mixes list/detail queries with multiple `collect()` and `Promise.all()` fan-outs, including full-table reads in `getMortgageCollectionOperationsSummary` at `798-910`.
     - `convex/payments/collectionPlan/workout.ts:362-387` loads non-cancelled entries by querying every status separately and flattening the results.
     - `convex/payments/collectionPlan/workout.ts:634-1045` embeds validation, rejection mapping, audit logging, supersession checks, and writes in a single activation path.
     - `convex/payments/collectionPlan/workout.ts:1048-1272` repeats public/internal wrapper plumbing four times for create/activate/complete/cancel.
   - Why this is a maintainability problem:
     - The admin surface is doing query selection, filtering, read-model projection, and transport shaping all in one place.
     - The workout surface repeats “build result, audit it, return it” and “public wrapper plus internal wrapper” patterns instead of using a lifecycle runner abstraction.
     - These files are already large enough that small features will encourage copy-paste over reuse.

3. **High**: Test infrastructure is fragmented across multiple helper layers, so new tests have too many ways to bootstrap the same world.
   - Evidence:
     - `src/test/auth/helpers.ts:16-42` defines one ad hoc module-loading strategy for `convex-test`, and `src/test/auth/helpers.ts:127-137` registers components inline.
     - `convex/test/registerAuditLogComponent.ts:1-101` defines a second component-registration path.
     - `convex/test/moduleMaps.ts:1-...` maintains a large manual module map in parallel with glob-based loading.
     - `src/test/convex/payments/helpers.ts:13-36` re-exports a third bootstrap layer via `createGovernedTestConvex`.
     - `src/test/convex/payments/helpers.ts:54-165` and `205-480` add payment-specific seeders instead of building on a single canonical test kit.
     - `src/test/convex/payments/reliabilityHarness.ts:14-44` and `228-1049` then build a fourth, bespoke reliability world on top.
   - Why this is a maintainability problem:
     - There is no single “how tests bootstrap Convex in this repo” answer.
     - The same concepts are repeated at different layers: component registration, identity setup, seed users, seed payment state, drain timers, source objects.
     - This makes test additions slower and increases the odds that one helper path drifts from another.

4. **Medium**: Public/internal transfer mutation pairs are still duplicated instead of sharing one orchestration core.
   - Evidence:
     - `convex/payments/transfers/mutations.ts:130-256` and `436-520` each implement create-transfer flow with overlapping idempotency, org resolution, and insert logic.
     - `convex/payments/transfers/mutations.ts:307-431` and `525-641` each implement initiate-transfer flow with overlapping query/load/validate/provider/transition logic.
   - Why this is a maintainability problem:
     - The file has already extracted `validateTransferCreationInput`, which is good, but the majority of the orchestration is still duplicated.
     - The next transfer-side change will need to be kept in sync across both public and internal paths.
     - This is exactly the kind of “same algorithm, different actor/source wrapper” case that should collapse into a single runner plus thin adapters.

5. **Medium**: The AMPS UI repeats the same list/detail/filter/action patterns across routes and dialogs instead of introducing reusable admin surface primitives.
   - Evidence:
     - `src/routes/demo/amps/-collection-plan.tsx:22-80` and `82-250` implement selection state, counts, filters, and a master-detail layout.
     - `src/routes/demo/amps/-collection-attempts.tsx:18-77` and `79-239` repeat the same page structure with different field names.
     - `src/routes/demo/amps/-rules.tsx:22-73` and `75-234` repeat the same selection/filter/detail pattern again.
     - `src/routes/demo/amps/-mortgages.$mortgageId.payments.tsx:35-100` manually derives route state from the pathname and then composes many one-off detail cards.
     - `src/components/demo/amps/dialogs.tsx:263-346`, `350-455`, `459-597`, `600-766`, and `768-1213` repeat `open`/`busy`/toast/submission/reset scaffolding across five dialog components.
   - Why this is a maintainability problem:
     - The AMPS workspace is already a mini-admin system, but it still behaves like five one-off pages.
     - The route files are very similar structurally; the code is not.
     - The dialog file is a monolith because there is no shared `ActionDialog`/`MutationDialog` primitive and no shared hook for the list-detail surface pattern.

6. **Medium**: Legacy compatibility logic in cash-ledger integrations still carries branch complexity that likely no longer pays for itself.
   - Evidence:
     - `convex/payments/cashLedger/integrations.ts:22-64` still accepts and normalizes a `LegacySource` shape.
     - `convex/payments/cashLedger/integrations.ts:1262-1336` implements four-tier payout detection specifically to catch legacy payout shapes.
     - `convex/payments/cashLedger/integrations.ts:1380-1622` keeps the reversal cascade broad enough to support those legacy fallbacks.
   - Why this is a maintainability problem:
     - The branch is already deleting older payment abstraction files elsewhere, but this file still preserves compatibility paths and ambiguity fallbacks.
     - In a greenfield repo with no production migration burden, keeping both old and new shapes increases the hardest file’s complexity without the usual operational justification.
     - The broad fallback at `1325-1336` is especially expensive cognitively because it encodes “best effort” matching rules in the most compliance-sensitive ledger path.

7. **Medium**: `convex/crm/recordQueries.ts` is oversized and duplicates its own link-resolution abstraction.
   - Evidence:
     - The file already has shared link helpers in `convex/crm/recordQueries.ts:319-385`.
     - `convex/crm/recordQueries.ts:702-765` re-implements the same link resolution logic inline inside `getRecord`.
   - Why this is a maintainability problem:
     - This is not just a big file; it is a big file with local duplication inside the same module.
     - The query/read-model layer is already handling EAV assembly, filtering, sorting, pagination, native-object bridging, and link hydration; duplicated link code is a signal that the module boundary is too wide.

8. **Low**: E2E auth setup is duplicated in two near-identical files.
   - Evidence:
     - `e2e/auth.setup.ts:4-34`
     - `e2e/amps/auth.setup.ts:4-17`
   - Why this is a maintainability problem:
     - Both files define the same `requireEnv` helper and the same “login then write storage state” shape.
     - This is a small issue, but it is easy copy-paste debt to remove and it will only grow if more e2e personas are added.

## Quick Wins

- Extract a single `withActorContext` or `runAdminActionBridge` helper for the repeated public/internal mutation wrappers in `collectionPlan/workout.ts` and `payments/transfers/mutations.ts`.
- Introduce a small `createAuthStorageState` helper and reuse it from both `e2e/auth.setup.ts` and `e2e/amps/auth.setup.ts`.
- Replace the duplicated inline link resolver in `convex/crm/recordQueries.ts:727-758` with the existing `resolveLinkedRecords` / `loadLinksForReference` helpers.
- Add a reusable `useSelectableSurface` hook for the repeated “selected row id + reset if missing” logic in the AMPS route files.
- Add a reusable dialog submit wrapper for the repeated `open` / `busy` / `toast` / reset flow in `src/components/demo/amps/dialogs.tsx`.
